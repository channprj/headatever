# Headatever

> **`head.yymmdd.patch`** — 사람이 직접 관리할 부분은 **head** 하나뿐인, 달력 기반 SemVer 호환 버전 체계. 나머지는 날짜와 하루치 카운터일 뿐입니다.

[English](./README.md)

**Headatever** 는 [HeadVer](https://github.com/line/headver) 의 "하루 단위(day‑precision)" 형제입니다. *직접 관리할 가치가 있는 숫자는 head 하나뿐* 이라는 HeadVer 의 핵심 철학은 그대로 유지하되, HeadVer 의 ISO **연‑주차(year‑week)** 필드를 **달력 날짜**(`yymmdd`) 로 바꿨습니다.

HeadVer 는 주차 단위면 충분하다고 보았습니다. Headatever 는 그 대신 정확한 날짜를 찍습니다 — **whatever**, 날짜가 더 다루기 쉽고, 잘못 읽을 일이 없으며, ISO 주차 번호의 자잘한 예외에서 자유롭기 때문입니다. 이름은 **Head**Ver + Cal**Ver** + "what**ever**" 를 합친 말입니다.

---

## 요약 (TL;DR)

- **포맷:** `head.yymmdd.patch`, 릴리스·태그는 `vhead.yymmdd.patch`.
- **`head`** 만 사람이 손대는 유일한 필드입니다. 호환성을 깨는 변경이나 마일스톤 릴리스에서 올립니다.
- **`yymmdd`** 는 릴리스 날짜에서 자동 생성되고, **`patch`** 는 매일 `0` 으로 초기화되는 카운터입니다.
- 모든 유효한 Headatever 버전은 동시에 유효한 [Semantic Version](https://semver.org) 이며 **정렬 결과도 동일** 합니다 — 기존 SemVer 도구를 그대로 쓸 수 있습니다.
- "이건 minor 인가 patch 인가?" 를 두고 다툴 일이 사라집니다. 달력과 카운터가 알아서 정합니다.

## 포맷

```
  head  .  yymmdd  .  patch
   │         │          └── 0부터 시작하는 하루치 릴리스 카운터 (0, 1, 2, …)
   │         └───────────── 2자리 연도(2000년대) + 0으로 채운 월 + 0으로 채운 일
   └─────────────────────── 수동으로 올리는 릴리스 라인 (≥ 0)
```

| 구성 요소  | 출처     | 예시     | 의미                                        |
| ---------- | -------- | -------- | ------------------------------------------- |
| `head`     | **수동** | `1`      | 릴리스 라인. 호환성 파괴/마일스톤 때 올림   |
| `yymmdd`   | 자동     | `260424` | 달력 날짜 `2026-04-24`                      |
| `patch`    | 자동     | `0`      | 그날의 첫 번째 릴리스 (0부터 시작)          |

> **`1.260424.0`** → head `1`, **2026‑04‑24** 릴리스, 그날의 `0`번째(첫) 릴리스.
> 정규 버전: `1.260424.0` · Git 태그: `v1.260424.0`

## 필드

### `head`

- **음이 아닌 정수**(`≥ 0`), **앞자리 0 금지**(단, 단독 `0` 은 허용).
- 호환성을 깨는 변경이나 마일스톤 릴리스를 알리기 위해 **수동** 으로 올립니다 — SemVer 의 `major`, HeadVer 의 `head` 에 해당합니다.
- 프로젝트 수명 동안 **단조 비감소(monotonically non‑decreasing)**, 절대 초기화하지 않습니다.
- `0` 은 **초기 개발(initial development)** 을 뜻합니다 — SemVer 의 `0.x`, HeadVer 의 0부터 시작하는 head 와 동일합니다. `0.*` 라인은 안정성을 약속하지 않습니다. 첫 안정/마일스톤 릴리스에서 `1` 로 올리세요.
- 기존 소프트웨어에 Headatever 를 도입할 때는 `head` 를 현재 major 버전 이상에서 시작하세요.

### `yymmdd`

릴리스 날짜로 만든 6자리 정수 하나입니다.

- **`yy`** — 연도의 끝 두 자리, **2000년대** 로 해석 (`26` → `2026`). 범위 `00`–`99`.
- **`mm`** — 0으로 채운 월, `01`–`12`.
- **`dd`** — 0으로 채운 일, `01`–`31`.

순수한 정수이므로 달력에 따라 단조 증가합니다:
`260424` < `260425` < `261231` < `270101`.

> **타임존:** 날짜는 문서화된 **단일 타임존** 에서 계산해야 합니다(**UTC 권장**). 그래야 빌드 머신·기여자 사이에서 하루치 카운터가 일관되게 유지됩니다.

### `patch`

- **음이 아닌 정수**(`≥ 0`), **앞자리 0 금지**(단, 단독 `0` 은 허용).
- **0부터 시작하는 하루치 카운터**: 해당 `yymmdd` 의 첫 릴리스는 `0`, 같은 날 추가 릴리스마다 `+1`.
- `yymmdd` 가 바뀌면 반드시 `0` 으로 초기화됩니다.
- 보통 빌드/릴리스 파이프라인이 **자동** 으로 증가시키므로, 수동 단계를 빠뜨려도 버전이 중복될 일이 없습니다.

## 규칙

아래의 **MUST**, **MUST NOT**, **SHOULD**, **MAY** 는 [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) 를 따릅니다.

1. 버전은 `head.yymmdd.patch` 형식에 **반드시(MUST)** 맞아야 합니다.
2. `head` 는 `≥ 0` 이어야 하고(**MUST**), 앞자리 0을 **포함하면 안 됩니다(MUST NOT)** (단, 단독 `0` 은 허용).
3. `yymmdd` 는 구조적으로 유효한 날짜여야 합니다(**MUST**): `mm` 는 `01`–`12`, `dd` 는 `01`–`31`. 실제 달력 유효성(예: `260230` 거부)은 코드에서 검증하는 것이 **좋습니다(SHOULD)**.
4. `yy` 는 2000년대 연도로 **반드시(MUST)** 해석합니다 (`26` → `2026`).
5. `patch` 는 `≥ 0` 이어야 하고(**MUST**), 그날 첫 릴리스에서 `0` 이어야 하며, 같은 날 추가 릴리스마다 정확히 `1` 씩 증가하고, `yymmdd` 가 바뀌면 `0` 으로 초기화되어야 합니다.
6. 정규 버전은 저장소 루트의 **`VERSION`** 파일에 있어야 하며(**MUST**), 앞에 `v` **없이** 저장합니다.
7. 릴리스 태그를 만들기 전에 `VERSION` 을 **반드시(MUST)** 갱신합니다.
8. Git 릴리스 태그는 `v{VERSION}`(즉 `vhead.yymmdd.patch`)이어야 하며(**MUST**), 수동 태그나 파이프라인 입력값도 이와 정확히 일치해야 합니다.
9. 버전은 SemVer 의 사전 릴리스 / 빌드 메타데이터 접미사를 **붙여도 됩니다(MAY)** (예: `1.260424.0-rc.1`, `1.260424.3+sha.9f2a`). 그래도 유효한 Headatever 버전입니다.

## 단일 진실 공급원 (Source of truth)

- 정규 버전은 저장소 루트의 **`VERSION`** 파일에 둡니다.
- `VERSION` 은 앞의 `v` **없이** 버전을 저장합니다.
- Git 릴리스 태그는 **`v{VERSION}`** 을 사용합니다.

| 산출물        | 값             |
| ------------- | -------------- |
| `VERSION`     | `1.260424.0`   |
| Git 태그      | `v1.260424.0`  |
| CLI / 표시    | `1.260424.0`   |

## 정렬과 SemVer 호환성

Headatever 는 SemVer 의 세 자리 숫자 구조를 그대로 빌려옵니다. 우선순위는 `head`, `yymmdd`, `patch` 순서로 각각 **숫자 비교** 하여 결정됩니다.

```
0.260424.0  <  1.260424.0  <  1.260424.1  <  1.260425.0  <  1.261231.9  <  2.260101.0
```

- `yymmdd` 가 단조 증가하는 정수이므로, SemVer 우선순위 규칙만으로 **날짜순 정렬이 그냥 됩니다** — 별도 비교기가 필요 없습니다.
- 따라서 (`yymmdd` 에 앞자리 0이 없는 `2010` 년 이후의) 모든 Headatever 버전은 **유효한 Semantic Version** 이며 정렬 결과도 동일합니다. npm, Cargo, pip, Go 모듈, 일반 정렬 루틴 등 기존 도구가 **수정 없이** 처리합니다.
- 관계는 한 방향입니다: **모든 Headatever 버전은 SemVer 버전이지만, 모든 SemVer 버전이 유효한 Headatever 버전은 아닙니다.** Headatever 는 가운데 필드를 실제 날짜로, 마지막 필드를 하루치 카운터로 추가 제약하기 때문입니다.

## 검증 (Validation)

구조 검증용 정규식 (ECMAScript 계열):

```regex
^(?<head>0|[1-9]\d*)\.(?<yymmdd>\d{2}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01]))\.(?<patch>0|[1-9]\d*)$
```

> 이 정규식은 **구조**(월 `01`–`12`, 일 `01`–`31`)만 검사합니다. `260230`(2월 30일) 같은 실제로 불가능한 날짜는 **걸러내지 못하므로** 코드에서 검증하세요.

## 왜 Headatever 인가?

세 가지 문제, 세 가지 의도적 선택:

1. **SemVer 는 논쟁을 부른다.** 이 변경은 "minor" 인가 "patch" 인가? 논쟁은 끝이 없고, 그 답이 다운스트림에 실제로 중요한 경우는 드뭅니다. Headatever 는 아예 질문을 없앱니다 — 날짜와 하루치 카운터는 자동으로 정해집니다. **사람이 정하는 건 `head` 하나뿐**, 정말로 중요한 신호 단 하나, "이번 릴리스는 다르다" 를 위해 남겨둡니다.

2. **CalVer 는 호환성 신호를 잊는다.** 순수한 달력 버전은 *언제* 를 전하지만, *업그레이드할 때 조심해야 하는지* 는 말해주지 않습니다. Headatever 는 바로 그것을 위해 SemVer 식 선두 `head` 를 유지합니다.

3. **HeadVer 는 주 단위로 반올림한다.** HeadVer 의 `yyww` 는 간결하지만, ISO 주차 번호에는 거친 구석이 있습니다 — 53주짜리 연도, 로케일에 따라 다른 주 시작 요일, 그리고 사람이 한눈에 날짜로 되돌릴 수 없는 숫자. Headatever 는 날짜 전체를 찍습니다: 모호하지 않고, 사람이 읽기 쉬우며, 정렬도 자명합니다.

요컨대 Headatever 는 **CalVer 의 날짜 코어 + SemVer 의 head + 하루치 빌드 카운터** 를, 전체가 여전히 유효한 Semantic Version 이 되도록 배치한 것입니다.

| 특성                          | SemVer            | CalVer              | HeadVer            | **Headatever**       |
| ----------------------------- | ----------------- | ------------------- | ------------------ | -------------------- |
| 포맷                          | `major.minor.patch` | 다양 (예: `YYYY.MM.DD`) | `head.yyww.build` | `head.yymmdd.patch` |
| 사람이 관리하는 필드          | major, minor, patch | 보통 없음         | head               | **head 하나뿐**      |
| 날짜 정밀도                   | 없음              | 일 / 월 / 주        | ISO 주차           | **달력 일(day)**     |
| 자동 생성 필드                | 없음              | 날짜                | yearweek, build    | yymmdd, patch        |
| 호환성 파괴 신호              | major             | —                   | head               | head                 |
| SemVer 로 정렬됨              | 예                | 경우에 따라         | 예                 | **예**               |
| "minor 냐 patch 냐" 논쟁      | 있음              | 없음                | 없음               | **없음**             |

## FAQ

**HeadVer 의 주 단위 대신 왜 날짜 단위인가요?**
전체 날짜는 모호하지 않고 사람이 읽기 쉽습니다. 빌드가 *정확히 언제* 나갔는지 항상 알 수 있고, ISO‑8601 의 53주차나 로케일 주차 문제도 없습니다. HeadVer 는 주차면 충분하다고 보았고, Headatever 는 그냥 날짜를 찍습니다 — *whatever*, 그게 더 다루기 쉬우니까요.

**하루에 두 번 이상 릴리스하면요?**
`patch` 를 올리세요: 그 `yymmdd` 안에서 `…​.0`, `…​.1`, `…​.2`, … 다음 날이 되면 `0` 으로 초기화됩니다.

**"하루" 의 기준 타임존은요?**
하나를 정하고 문서화하세요. 머신·타임존 간 카운터 일관성을 위해 **UTC 를 권장** 합니다.

**`head` 도 초기화되나요?**
아니요. 프로젝트 수명 동안 단조 비감소입니다. `0` 은 아직 첫 마일스톤 릴리스를 내지 않았다는 뜻일 뿐입니다.

**`-rc.1` 같은 사전 릴리스 태그나 `+sha.9f2a` 같은 빌드 메타데이터를 쓸 수 있나요?**
네. SemVer 접미사가 허용되며, 그래도 SemVer 호환을 유지합니다.

**기존 프로젝트에 어떻게 도입하나요?**
`head` 를 현재 major 버전 이상에서 시작하고(신규 프로젝트라면 초기 개발 단계를 뜻하는 `0` 에서 시작), `VERSION` 을 그에 맞게 설정한 뒤, 그때부터 파이프라인이 `yymmdd` 와 `patch` 를 채우게 하세요.
